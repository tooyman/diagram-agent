meta-diagram-svg

Generates good-looking logical architecture diagrams from declarative YAML. The LLM only writes schema-valid metadata — layout and rendering are deterministic.

Why this exists

Asking an LLM to produce a diagram directly — either drawio XML, SVG coordinates, or an image — fails reliably: bad alignment, lines overlapping shapes, ghost components, edges that don’t connect to anything. Multi-round visual verification loops make it worse, not better, because the model has no objective gradient to follow.

This skill removes the LLM from the visual loop entirely. It produces only a structured description (YAML); a deterministic pipeline does the geometry, validates it numerically, and emits SVG.

Pipeline

1
LLM
Emits YAML matching the schema. No coordinates.
2
Pydantic
Validates structure & references. Rejects bad input early.
3
ELK (Node.js)
Layered + orthogonal layout, nested containers.
4
Validator
Overlaps, edge-through-node, crossings, label hits. Retries with new params on hard fail.
5
SVG
Per-type shapes, wrapped multi-line labels, design tokens.
6
HTML viewer
Mousewheel zoom, drag pan, Fit button (Cmd/Ctrl+0).

Stage 4 is the key insight. The retry loop has a deterministic gradient (geometry checks return a number), so it actually converges — unlike asking the LLM to “look at the picture again and fix it.”

title: "Order processing (simple)"

groups:                          # nestable via `parent`
  - id: app
    name: "Application tier"
  - id: data
    name: "Data tier"

components:
  - id: api
    name: "Order API"
    group: app
    type: server                 # see vocab below
    lifecycle_status: unchanged  # unchanged | updated | new

  - from: api
    to: db

Quick start

The skill is self-contained. One-time install:

.claude/skills/meta-diagram-svg/setup.sh

Creates a bundled .venv/ and runs npm install inside the skill folder. Requires python3 and node on PATH.

Generate a diagram (paths are relative to your CWD):

.claude/skills/meta-diagram-svg/run.sh nleads_metadata.yaml -o output/nleads.svg

Two files are written:

  • output/nleads.svg — the raw SVG (scales in any browser)
  • output/nleads.html — viewer with pan/zoom (scroll, drag, Cmd 0)

Validator behavior

Geometry checks run after layout. Hard failures — node overlap, edge crossing a node, group overflow, excessive edge crossings — trigger automatic retries with adjusted ELK parameters (up to 5). The retry loop is purely deterministic; the LLM is never involved.

What the LLM should and shouldn’t do

Do

  • Read user input (prose / notes / table) and extract components, groups, connections.
  • Pick type values from the closed vocabulary.
  • Use stable id values (snake_case, no spaces).
  • Reference existing components by id in from/to.
  • Run the CLI and report the output paths plus any validator warnings.

Don’t

  • Produce SVG, drawio XML, coordinates, or anything pixel-related.
  • Visually verify the output and try to “fix” it — that’s the validator’s job.
  • Invent new type / lifecycle_status / status values.
  • Add cloud-provider icons or vendor-specific shapes (this is a logical diagram tool).
